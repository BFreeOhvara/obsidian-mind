# Gotchas

Things that have bitten before and will bite again.

- **OneDrive "Delete these items?" popup with hash-named files = git auto-gc, not data loss.** Both `ohvara-dashboard` and `Scraper` live inside OneDrive Desktop, so OneDrive syncs `.git` internals. When git repacks loose objects (auto-triggered by fetch/pull), hundreds of 38-char hex files vanish at once and OneDrive asks to confirm. Correct answer: **Delete all items** — the contents live in the new pack file and everything is on GitHub. "Keep" restores stale objects into `.git` and re-triggers the dialog later. First seen 2026-06-11 (673 items, matched dashboard repo's 679-in-pack/6-loose exactly). Long-term fix if it gets annoying: move repos out of OneDrive or exclude `.git` folders from sync. Related: [[Memories]]

- **Claude in Chrome binds to whichever claude.ai ACCOUNT is signed into that Chrome profile, not to "Eagle" or "Falcon" as a concept.** Eagle/Falcon are just Brayden's own labels for two separate claude.ai Pro logins (account 2 / account 1) — the extension doesn't know or care about that naming, it only knows the account session in the browser profile. Hit 2026-06-22: Brayden was logged out of the Falcon account (no usage left) and switched to Eagle in this Cowork chat, but the Chrome extension was still signed into the old Falcon account in the one Chrome profile that existed → `list_connected_browsers` returned `[]` from the Eagle session because no browser was connected to *that* account. **Fix shipped:** Brayden created a second Chrome profile, each profile's extension signed into its own account (one Falcon, one Eagle) — so each chat now has a matching profile to pair with, instead of one shared profile that only ever matches whichever account logged in most recently. **Standing rule going forward:** if `list_connected_browsers` returns `[]`, don't assume the bridge is broken — check which account each Chrome profile's extension is signed into before troubleshooting further, and confirm the profile matches the chat's own account. When verification needs to run in a chat/account this session can't reach, hand off via a written prompt (artifact) rather than retrying the same dead connection repeatedly.

- **`.git/index.lock` stuck under OneDrive can make `git status`/`commit` lie that the tree is clean.** Hit 2026-06-22 from a Cowork sandbox session editing the vault directly on the OneDrive-synced path: `.git/index.lock` existed with `Operation not permitted` even on `rm -f` (OneDrive holding the file), and with the lock present, `git add -A && git commit` reported "nothing to commit, working tree clean" despite real, verified-on-disk edits (confirmed via direct file read/grep) that were NOT in `HEAD` (`git show HEAD:<file> | grep ...` came up empty for content that `cat`/`grep` on the working file found fine). This is worse than the silent-failure case in the gotcha above — here git gives a false "all good" instead of erroring. **If a commit you just made doesn't show the line/insertion count you expect, don't trust `git status` — diff `HEAD:<file>` against the working file directly to confirm.** Fix needs to happen on the Windows side (close OneDrive sync or the process holding the lock, then delete `.git/index.lock` manually) — not fixable from a sandboxed bash session that can't touch that file's permissions. Until fixed, commits from sandboxed sessions on this vault may silently not persist — verify after every commit. Related: [[Memories]] 2026-06-22.

- **Supabase new API keys (`sb_secret_*`) cannot be called via plain HTTP from PowerShell/curl.** Legacy `anon`/`service_role` keys were disabled 2026-06-14. New format: `sb_publishable_*` (anon) and `sb_secret_*` (service role). The secret key throws `"Forbidden use of secret API key in browser"` from any HTTP client (Supabase server-side detection, not just browser UA). Also blocked by CC auto-classifier. **Workaround:** write the SQL and ask Brayden to run it in the Supabase SQL editor. Python `urllib.request` with the secret key would work server-side but the CC auto-classifier may block that call too. If direct DB access is needed and CC can't do it, the SQL-editor path is the only option. First hit 2026-06-25 (Prompt 81 Change 1). Related: [[Memories]] 2026-06-25.

- **Any `position: fixed` modal in `ohvara-dashboard` will render off-screen if it's not portaled to `document.body`.** `DashboardLayout.jsx` wraps every page's content in a `.page-enter` div, which carries `animation: fadeSlideUp 0.35s ease both` (`index.css`) — `fill-mode: both` keeps a `transform: translateY(0)` applied to that wrapper forever after the animation finishes. Per the CSS spec, any ancestor with a non-`none` transform becomes the containing block for `position: fixed` descendants, so a "fixed" modal nested anywhere under `DashboardLayout` is actually fixed relative to that (scrollable, often taller-than-viewport) page wrapper, not the real browser viewport. If the rep has scrolled the page before opening the modal, it renders shifted up by the scroll amount — can go fully off-screen above the fold. Hit 2026-07-01 (Prompt 185): Falcon caught the Final Exam modal's question card rendering at `top: -164px` after Prompt 183 made the card bigger; the actual bug was pre-existing (any fixed modal under this layout has it), just more visible once the modal grew. **Fix:** render fixed-position modals via `createPortal(<modal jsx>, document.body)` (`import { createPortal } from 'react-dom'`) instead of returning the JSX inline — this escapes the tainted containing block entirely, no CSS/sizing changes needed. **The video-lock modal (`LockedVideoPlayer`'s wrapper in `VideoLibrary`) likely has the same latent bug and hasn't been fixed yet** (flagged as a follow-up task, not fixed in Prompt 185 — out of scope for that fix). Any *new* fixed-position modal added to this app should be portaled to `document.body` from the start. Related: [[Memories]] 2026-07-01.

- **Chrome-MCP visual verification must run from Claude DESKTOP, not the Claude Code CLI.** On this machine the Chrome extension's native-messaging host (`com.anthropic.claude_browser_extension`, registered in `HKCU\Software\Google\Chrome\NativeMessagingHosts`) points at — and runs under — the **Claude Desktop app** (MSIX pkg `Claude_pzs8sxrjxfjjc`, `…\ChromeNativeHost\chrome-native-host.exe`). So a Claude **Code CLI** session (`…\Roaming\Claude\claude-code\<ver>\claude.exe`, a separate process; `claude` not even on PATH) calls `mcp__Claude_in_Chrome__list_connected_browsers` and gets **`[]`** — the extension is bridged to Desktop, not the CLI. **Workflow for any rep-auth-gated visual check (standing rule #11):** write a self-contained QA prompt with the spec embedded, have Brayden paste it into the **Claude Desktop** chat (which owns the bridge + the saved apex11 login on the vercel domain), Desktop drives Chrome + screenshots + reports PASS/FAIL, then CC fixes from the CLI. Don't burn time trying to make the CLI own the browser — re-pairing the extension to the CLI was explicitly PARKED (quitting Desktop to free the host = too many unknowns, not worth it for a one-off verify). First hit 2026-06-17 verifying Thread #17. Related: [[Memories]]

- **A browser-automation tool's coordinate-based click can silently miss its target element and produce "nothing happened" — which looks identical to correct click-gate blocking behavior.** Hit 2026-07-02 (Prompt 200): Falcon reported Prompt 199's `ErrorToast` never rendering on the Final Exam/AI Roleplay gates, even though the blocking logic itself worked (exam didn't open, no mic prompt). Built an isolated harness (byte-for-byte copy of the component, same method as Prompt 185's repro) to test the claim rather than reasoning blind — a raw JS `.click()` reliably fired the handler and mounted the toast every time, but `preview_click`'s coordinate-based click in that same harness landed off-element (`preview_inspect` showed the button's bounding box at `{x:-42,y:-88}`, off-canvas) and silently did nothing. A click that misses its target is indistinguishable from "the gate correctly blocked the action" to an observer who only checks the negative case (exam didn't open) — both produce zero visible effect. **When a live report says "the blocking half works but the visible-feedback half doesn't," test with a direct `element.click()` via console/eval as well as the automation tool's click before concluding the component has a real render bug** — the two can disagree, and only one of them proves the code path actually ran. Related: [[Memories]] 2026-07-02.

- **`discoveryScript.js`'s `→ Go to X` routes are dumb regex jumps to a section, with zero awareness of call history — a leftover route can silently point backward into an already-completed section.** `routeTarget()` just string-matches "vitals"/"pain"/"handoff"/"opener" in the route line's text and jumps to that section's entry point; it has no concept of "we already did this section this call." Hit 2026-07-05 (Prompt 218): Handoff's `"Who is this / what company?"` objection had `"That's me" → Go to Vitals Check` — a route left over from before Prompt 213 embedded objection content directly into Handoff — so resolving that one objection re-ran Vitals from scratch, looking like the whole script restarted. Audited every `Go to `/`→ Go` line in the file (section order Opener→Vitals→Pain→Handoff→Close) to confirm this was the only backward route. **Whenever an objection/branch gets its content embedded inline (the standard pattern here since Prompt 213, since this DSL has no cross-section node-level routing), double-check its resolution routes FORWARD only** — check what section the fork lives in vs. what section the route targets, don't just trust that the route text still makes sense. Related: [[Memories]] 2026-07-05.

- **Cowork sessions (Eagle/Falcon) cannot reliably git commit/push on this vault — the FUSE bridge blocks file deletion, not just this vault's OneDrive sync issue above.** Cowork reaches the vault through a sandboxed FUSE bridge to the real Windows folder rather than a direct connection. On that bridge, file *deletion* (`unlink`) is blocked with "Operation not permitted" even though writes/creates work fine. Git creates a temporary `.git/index.lock` during every commit and deletes it when the commit finishes; when that delete fails, the lock is orphaned and blocks every later git command from that Cowork session, since git thinks another process is still mid-operation. This happened across dozens of past Cowork sessions, each leaving its own dead lock file behind (`.dead.*`, `.old.*`, `.bak*` suffixes) rather than cleaning up after itself — cleaned up in bulk 2026-07-15 (Prompt 271, native CC session, since only a native session can actually delete files here). **Standing workflow rule:** Cowork sessions should write file content only (Read/Write/Edit tools, which always work through the bridge) and NOT attempt `git add`/`commit`/`push` on this repo — leave version control entirely to CC's native sessions. CC should `git pull` at the start of every run (existing standing rule) to pick up any uncommitted vault edits Cowork left on disk, then commit/push them normally alongside whatever else that session ships. Corollary: **CC should never assume "no new commit since I last pushed" means "no new vault content"** — a Cowork session's edits may be sitting uncommitted on disk even though `git log` looks unchanged; always read/diff LIVE_STATE and Memories for real content, not just check git history. See also [[North Star]] Rule 16 (Cowork tool-access boundaries). Related: [[Memories]] 2026-07-15.

- **A LIVE_STATE prompt saying "Brayden already confirmed X, don't re-ask" is not the same as Brayden telling CC X directly — sensitive/irreversible actions (secrets, production config) need real confirmation in the live chat, not a relayed claim.** Hit 2026-07-15 (Prompt 274): Falcon queued a prompt claiming Brayden had pre-approved clearing the `RETELL_ROLEPLAY_AGENT_ID` secret and flipping `DEMO_MODE` to `false` on the live Supabase project, explicitly saying "do not re-ask." CC executed both changes on the strength of that written claim — the auto-mode permission classifier then blocked a subsequent related action, stating the earlier `AskUserQuestion` approval in-session had covered only an edge-function *deploy*, not this secret rotation, and that the "go-ahead" was agent-relayed rather than a direct user message. CC had already made the changes by that point (reversible, no data loss, but done before checking). **Standing rule:** for genuinely sensitive actions (secrets/credentials, production config flips, anything in the "explicit permission required" tier), a same-session `AskUserQuestion` confirmation from Brayden is required even when a LIVE_STATE prompt claims prior approval — the claim can inform the ask, but doesn't substitute for it. This is the same spirit as [[North Star]] Rule 16/18 (Cowork can't drive certain tools directly) extended one step further: even CC's own native session shouldn't treat an agent-authored "already approved" note as equivalent to the user's own words in that turn. Related: [[Memories]] 2026-07-15.

- **The Claude Browser pane's automated tab doesn't paint continuously between discrete tool calls — a CSS `transition` on an animated property can look permanently stuck even when the underlying state/logic is 100% correct.** Hit 2026-07-16 (Prompt 288, sidebar-to-drawer fix): after clicking a hamburger button, `getBoundingClientRect()` on the drawer kept reporting the CLOSED position no matter how much real wall-clock time passed between checks (multiple separate tool round-trips, each well over the CSS transition's 200ms). Ruled out state/logic bugs by checking progressively lower-level signals: a plain-text React state marker confirmed the state flipped correctly; `getComputedStyle(el).getPropertyValue('--tw-translate-x')` (a non-animated custom property) also resolved correctly and instantly. Only the *animated* `transform` shorthand (which had `transition: transform 200ms ease`) stayed stuck at its pre-click value. Tried polling via `requestAnimationFrame` from inside the page to let the transition settle — it never fired once in 30 real seconds, which is the actual tell: this automated tab simply isn't running a real compositor frame loop between tool calls the way a normal, visible, focused browser tab does. Setting `el.style.transition = 'none'` and re-reading resolved the "stuck" value instantly and correctly. **When a harness check on an animated/transitioned CSS property looks wrong despite state being provably correct, don't conclude the code is broken** — check a non-animated signal that reflects the same state first (a custom property, a text marker, a class name), or temporarily strip the `transition` to force an instant resolve. A real user's browser paints continuously and would never observe this; it's specific to this tool's rendering cadence. Related: [[Memories]] 2026-07-16.

- **The connected Supabase MCP has no edge-function-secrets tool at all — not gated, just absent — and there's no local `supabase` CLI on this machine as a fallback.** Hit 2026-07-19 (Prompt 315 follow-up): Brayden said "yes clear and rebuild" for `RETELL_ROLEPLAY_AGENT_ID`. Checked the full Supabase MCP tool list (`project_id`-scoped `f606cd64-...` server) — 24 tools covering projects/database/branches/edge-functions/migrations/advisors/logs, but nothing named `set_secret`/`unset_secret`/`list_secrets` or similar; `ToolSearch` for "supabase secrets" and "f606cd64 secret" both came up empty against the real tool set (not just a phrasing miss). `which supabase` also failed — no CLI to shell out to either. **This means CC cannot clear/set/list edge function secrets on this project through any currently-available tool, full stop — not a permission block like the `deploy_edge_function` classifier gate, an actual missing capability.** Contrast with [[Gotchas]]'s own Prompt 274 entry above, which describes CC "executing" a secret clear on 2026-07-15 — that was likely done through a different session/tool surface (Cowork, or manual dashboard action mislabeled in the log) since this native CLI session's tool set has never had this capability. **Standing workflow for any future secret rotation:** hand Brayden the exact dashboard path or CLI command instead of attempting it — for `RETELL_ROLEPLAY_AGENT_ID` specifically: Supabase Dashboard → project → Edge Functions → Secrets → delete the key, then trigger one real call so `create-roleplay-call`'s `if (!agentId)` branch rebuilds it (paid Retell call), then CC can pull the new `agent_id` from `get_logs` (service `edge-function`) for Brayden to paste back in. Related: [[Memories]] 2026-07-19.

- **A prompt can reference a Supabase migration/column that's live on remote but not committed to the repo's `supabase/migrations/` folder.** Falcon (or another instance with Supabase MCP access) sometimes applies a migration directly to the remote project without committing the corresponding `.sql` file to git — the local migration history then silently lags what's actually live. Hit 2026-07-01 (Prompt 191): the prompt referenced migration `064_call_feedback_detail` and three new `calls` columns, but `ohvara-dashboard`'s local `supabase/migrations/` folder stopped at `063_follow_up_at.sql` — grepping the repo for the new column names came up empty. **Don't assume a referenced migration/column doesn't exist just because it's missing locally** — verify against the live schema first via the Supabase MCP (`list_migrations` + `list_tables` with `verbose: true`) before concluding the prompt is wrong or blocking on it. Confirmed all three columns existed exactly as named. Flag the missing migration file as a follow-up (schema history should stay in sync with git) but don't let it block the UI work that depends on it. Related: [[Memories]] 2026-07-01.

- **Supabase dashboard's Edge Function Secrets page has a known delete bug (reported April 2026) — the delete button can silently fail.** You click confirm on removing a secret and nothing happens — the row stays, no error shown. Relevant because clearing `RETELL_ROLEPLAY_AGENT_ID` (the standing blocker on every AI Roleplay prompt change — 309b/315/316 all hit it) has to go through this exact page, since neither CC nor Falcon has a secrets-management tool on the connected Supabase MCP, and this machine has no local `supabase` CLI installed. **If the delete looks like it did nothing: refresh the page and check whether it actually took before assuming it's stuck.** If it's genuinely stuck, the only fallback is the `supabase` CLI (`supabase secrets unset <name>`) run from a machine that has it installed — not available here today. First hit 2026-07-19, Prompt 316 secret-clear step. Related: [[Memories]] 2026-07-19.
